## 1. 项目背景与核心定位

### 1.1 系统架构

本项目构建的是一个**大模型 + 小模型协作**的 Text-to-SQL 系统，服务于工程建设项目财务管理场景。系统架构如下：

```
用户自然语言提问
        ↓
大模型（云端）
  - 读取 Skills（含视图描述、字段说明、业务规则）
  - 完成 Schema Linking（确定相关视图和字段）
  - 将复杂查询分解为多个自然语言子任务
  - 为每个子任务附带相关 Schema 子集
        ↓
小模型（本地，0.5B 量级）—— 本项目的核心
  - 接收 [子任务描述 + Schema 子集] 作为输入
  - 并行生成 MySQL SQL 语句
  - 执行失败时接收错误信息，进行自我纠错
        ↓
执行结果汇总
        ↓
大模型生成综合性分析回答
```

### 1.2 小模型的核心价值

在此架构中，小模型承担的是**执行器**角色，其核心优势体现在两点：

- **并行性**：大模型分解出的多个独立子任务可以同时交给小模型处理，显著降低响应延迟
- **自我纠错**：系统架构支持 SQL 执行反馈回路，小模型可在执行失败后接收报错信息并修正

### 1.3 数据库背景

目标数据库为 MySQL，包含 14 个视图，覆盖工程建设项目财务管理的核心业务实体：

| 视图类别 | 视图名称 |
|---------|---------|
| 合同管理 | `v_project_contract`、`v_cost_contract` |
| 发票管理 | `v_receipt_invoice`、`v_payment_invoice`、`v_refund_invoice` |
| 资金管理 | `v_receive_payment`、`v_payment`、`v_refund` |
| 汇总视图 | `v_project_finance_summary`、`v_salesman_summary` |
| 基础数据 | `v_customer`、`v_party_company`、`v_subcon_company`、`v_user` |

## 2. 思考过程与方案演化

本章记录了从初始设想到最终方案的完整讨论轨迹，包括关键的思路碰撞与方向转变。

### 2.1 数据增强方案的初始设想与校正

**初始提案（四个维度）：**

最初提出的数据增强方案包含四个维度：Schema Linking 复杂性、DSL 到 SQL 的复杂映射、鲁棒性数据、以及自我纠错数据。

**关键校正一：子任务格式的确定**

讨论中确认子任务以**自然语言**形式呈现（而非结构化 DSL），理由是用户可以直观看到每个子任务的含义，提升系统可解释性。

**关键校正二：Schema Linking 的归属重新理解**

> 这是本次讨论中最重要的架构澄清。

初始方案中提出了基于 Embedding 的 Schema Linker，用于从 14 个视图中筛选出相关视图。然而在明确系统架构后，发现**大模型通过 Skills 机制已经完成了 Schema Linking**——大模型读取 Skills 后，会为每个子任务附带已筛选好的 Schema 子集再传给小模型。

这一澄清导致数据增强方案的核心调整：

| 调整前 | 调整后 |
|-------|-------|
| 训练小模型从全量 Schema 中找到相关视图 | 训练小模型在**给定**的 Schema 子集中精确映射字段 |
| 需要 Embedding 模型做预筛选 | 小模型只负责 SQL 生成，无需视图选择 |
| Schema Linking 是独立组件 | Schema Linking 由大模型+Skills 机制承担 |

"Schema Linking 复杂性"的训练目标因此重新定义为：在给定视图子集中处理**字段歧义**、**同名字段的表归属**、以及**业务概念到物理字段的映射**。

### 2.2 上下文窗口的硬约束分析

在讨论数据格式时，进行了一次关键的 token 预算估算：

```
完整 Schema（14 个视图 × 平均 25 字段）≈ 1400~2000 tokens
自然语言子任务                         ≈   50~150 tokens
SQL 输出                               ≈  100~300 tokens
─────────────────────────────────────────────────────────
合计                                   ≈ 1650~2450 tokens
```

0.5B 模型的上下文窗口通常为 2048~4096 tokens，这意味着**全量 Schema 注入在边界情况下会撑爆上下文**。这进一步验证了大模型预先筛选 Schema 的必要性，也确定了小模型接收的输入始终是精简后的 Schema 子集。

### 2.3 Schema 字段注释的混合粒度策略

在分析真实视图 Schema 后，发现字段分为两类：

- **普通字段**：命名清晰，如 `invoice_price_yuan`、`customer_name`，无需额外注释
- **特殊字段**：需要业务说明，典型例子：
  - `tax_rate: INTEGER`——以 ×10 整数存储，130 表示 13%，不注释则无法正确生成查询条件
  - `status: TINYINT`——枚举值含义不透明
  - `contract_status: VARCHAR`——枚举字符串集合未知（如 `'UN_DONE'`、`'DONE'`）

这一发现确立了训练数据的**混合粒度策略**：60% 的样本使用纯字段名（训练从命名推断语义），40% 使用带注释的字段（训练优先利用注释约束）。

### 2.4 并行架构的哲学讨论

在验证了微调效果后，讨论上升至编排架构层面，围绕以下核心问题展开：

> **子查询的数据依赖是否阻碍并行执行？**

通过分析得出结论：

**① ≤1 层依赖深度的子查询均可转换为 JOIN**

所有深度为 1 的子查询（包括相关子查询）都存在等价的 JOIN 改写：

```sql
-- 相关子查询（串行）
SELECT o.id FROM orders o
WHERE o.amount > (SELECT AVG(amount) FROM orders WHERE customer_id = o.customer_id)

-- 等价 JOIN（可并行处理各部分）
SELECT o.id FROM orders o
JOIN (
    SELECT customer_id, AVG(amount) AS avg_amt
    FROM orders GROUP BY customer_id
) t ON o.customer_id = t.customer_id
WHERE o.amount > t.avg_amt
```

这一结论的意义在于：**在深度 ≤ 1 的范围内，并行架构与子查询语义完全兼容**。

**② 深度 ≥ 2 时采用 DAG 调度**

当存在多层依赖时，将执行模型从"批量并行"升级为"DAG 层次并行"：

```
第一批（无依赖）：  [SQL-A]  [SQL-B]  [SQL-C]   ← 并行
                     ↓         ↓
第二批（依赖第一批）：  [SQL-D]    [SQL-E]        ← 并行
                              ↓
第三批：                    [SQL-F]               ← 依赖第二批
```

关键洞察：**每一层内部的 SQL 生成与自我纠错过程仍然可以并行进行**，只有跨层之间才需要等待。这与现代分布式计算框架（Spark、Airflow）的执行模型在哲学上完全一致。

### 2.5 数据飞轮的发现

在讨论评估体系时，自然引出了一个重要的产品设计洞察：

```
用户提问 → SQL 执行失败 → 自我纠错 → 执行成功
                ↓
    收集三元组：(问题 + Schema, 错误SQL + 报错, 修正SQL)
                ↓
         定期补充进训练集 → 重新微调
                ↓
    错误率下降 → 收集到更高质量、更难的边界案例
                ↓
              持续迭代...
```

生产环境天然形成数据飞轮：模型随使用时间持续改进，业务数据覆盖的查询分布是任何公开数据集都无法替代的。这被认为是系统长期竞争力的核心来源。

## 3. 当前模型的预期能力

### 3.1 训练数据构成

| 数据类型 | 数量 | 权重策略 |
|---------|------|---------|
| 业务种子数据（手工构造） | 55 条 | ×3 重复采样，高优先级 |
| 数据增强（同义词/粒度/干扰视图） | 157 条 | 全量 |
| 自我纠错三元组 | 16 条 | ×2 重复采样 |
| Spider 转换数据（MySQL 化） | 约 3000 条（配额） | 随机采样 |
| **最终训练集（去重后）** | **约 3500 条** | — |

### 3.2 预期掌握的能力

**SQL 结构能力：**
- 单表过滤（`WHERE`、`LIKE`、`IS NULL`）
- 聚合查询（`GROUP BY`、`HAVING`、`COUNT`/`SUM`/`AVG`）
- 多表关联（`JOIN`，包括 `LEFT JOIN`）
- 子查询与派生表
- 窗口函数（`ROW_NUMBER`、`RANK`、`LAG`）
- CTE（`WITH` 子句）

**MySQL 特性适配：**
- 正确使用 `DATE_FORMAT`、`DATE_SUB`、`DATEDIFF` 等日期函数（避免 SQLite 写法）
- `HAVING` 子句中使用聚合表达式而非别名
- `GROUP BY` 包含所有非聚合字段（兼容 `ONLY_FULL_GROUP_BY` 模式）
- 使用 `NULLIF` 防止除零错误

**业务语义理解：**
- `tax_rate` 的 ×10 存储单位（查询 13% 时写 `tax_rate = 130`）
- 预聚合视图的优先使用（查询项目汇总时用 `v_project_finance_summary`，而非 JOIN 原始表）
- 中文业务术语到物理字段的映射

**自我纠错能力（覆盖的错误类型）：**

| 错误类型 | 示例 |
|---------|------|
| `HAVING` 引用别名 | `HAVING total > 100` → `HAVING SUM(x) > 100` |
| `GROUP BY` 缺字段 | 补全非聚合字段 |
| 子查询未加别名 | 派生表添加 `AS t` |
| SQLite 日期函数 | `STRFTIME` → `DATE_FORMAT` |
| 除零错误 | 包裹 `NULLIF(..., 0)` |
| `LIKE` 缺通配符 | `'钢铁'` → `'%钢铁%'` |
| 窗口函数语法 | 补全括号 `OVER (PARTITION BY ...)` |
| `NULL` 比较写法 | `= NULL` → `IS NULL` |

## 4. 评估结果与暴露的缺陷

### 4.1 已完成的评估

在 Spider dev 集（约 1034 条）上进行了两层验证：

| 指标 | 基础模型 | 微调后模型 | 变化 |
|------|---------|----------|------|
| Syntax Valid Rate | 100% | 100% | — |
| Schema Valid Rate | 49% | **57%** | +8pp |

> **注**：Schema Valid Rate = 1 - Schema Error Rate。基础模型字段错误率 51%，微调后降至 43%。

### 4.2 错误分析

**错误类型的演变**是一个值得关注的信号：

| | 基础模型错误模式 | 微调后错误模式 |
|--|---------------|-------------|
| 典型错误 | 完全搞错查询表（从错误的表中取字段） | 正确识别了关联表，但字段归属仍有偏差 |
| JOIN 使用 | 很少使用 JOIN，倾向于错误子查询 | 开始正确使用 JOIN 结构 |
| 错误层次 | **表级别混乱** | **列级别混乱**（更精细的错误） |

这说明微调使模型**向正确方向进步了一层**——从不知道字段在哪张表，到知道表关系但字段细节仍不准确。

**一条误判的发现：**

```sql
WHERE country = `France`   -- 模型用反引号代替单引号
```

验证器将 `France` 识别为字段名并报错，但这实际上是引号类型错误，应归类为语法层面。这说明 Schema Valid Rate 可能略有虚低。

### 4.3 尚未完成的评估

**执行准确率（EX）尚未进行。**

执行准确率是最具说服力的指标：在 Spider SQLite 数据库上实际执行生成的 SQL 与金标准 SQL，对比结果集是否一致。评估脚本（`scripts/06_evaluate.py` Layer 3）已完成，等待下一阶段运行。

### 4.4 Spider 评估的局限性

Spider 评估测量的是**通用 SQL 能力**，训练数据是**特定业务领域的 14 个视图**。模型在 Spider 上的 8pp 提升是副产品。真正关键的评估是在实际业务 Schema 上——那里字段名和表结构是模型在训练中见过的，预期准确率会显著更高。

## 5. 未采用的技术路线

### 5.1 独立的 Embedding-based Schema Linker

**方案描述：** 训练一个独立的向量检索模块，对 14 个视图做向量化，通过子任务 Embedding 相似度检索出相关视图，再交给小模型。

**未采用原因：** 与系统架构中大模型+Skills 机制的功能直接冲突。大模型通过读取 Skills（其中包含视图描述、字段说明、业务规则）已经完成了视图筛选工作，额外的 Schema Linker 是冗余组件。

### 5.2 BIRD 完整训练集

**方案描述：** 下载 BIRD 数据集的完整训练集作为基础训练数据。

**未采用原因：** 训练集体量过大，下载成本高；BIRD 的核心优势（evidence 字段中的业务知识注释）对于 SQL 语法学习的边际贡献在 Spider 已足够的前提下不够显著。

### 5.3 BIRD Dev 集替代训练数据

**方案描述：** 用 BIRD Dev 集（约 1500 条）作为 BIRD 完整训练集的替代。

**未采用原因：** Dev 集是评测集，使用评测数据训练会污染评测结果，在方法论上不合适。

### 5.4 Unsloth 训练框架

**方案描述：** 使用 Unsloth + SFTTrainer 替代 LLaMA-Factory，以更简洁的代码实现训练。

**未采用原因：** 用户评估后认为其过于复杂，更倾向于 LLaMA-Factory 的配置文件驱动方式。最终回归 LLaMA-Factory，Unsloth 训练脚本保留在 `scripts/train_unsloth.py` 供参考。

### 5.5 全量 Schema 注入

**方案描述：** 将全部 14 个视图的完整 Schema 注入每条训练样本和推理 Prompt，不做预筛选。

**未采用原因：** Token 预算估算显示，全量 Schema 约 1400~2000 tokens，加上任务描述和输出，在边界情况下会超出 0.5B 模型的上下文窗口。且让小模型自己做 Schema Linking 会大幅增加任务难度，超出 0.5B 量级的合理能力边界。

### 5.6 Exact Match（EM）作为主评估指标

**方案描述：** 用生成 SQL 与金标准 SQL 的字符串相似度作为主要评估指标。

**未采用原因：** 同一查询意图存在多种等价 SQL 写法，字符串比较会产生大量误判（把语义正确但写法不同的 SQL 标记为错误）。最终采用执行准确率（EX）作为主指标，字符串比较方法完全废弃。

## 6. 未来技术路线

### 6.1 近期（下一阶段）

**全面执行准确率评估：** 利用 Spider 自带的 SQLite 数据库运行 Layer 3 评估，得到最具说服力的性能基准。评估脚本已就绪，使用 sqlglot 做 MySQL→SQLite 语法转换。

**业务域 held-out 测试：** 手工编写 10~20 条不在训练集中的新业务查询，测试模型在真实场景下的表现。这是 Spider 评估无法替代的。

**自我纠错能力评估：** 利用现有 `eval_results.jsonl` 中的错误预测作为测试用例，在 SQLite 上执行错误 SQL 获取真实报错，交给模型修正，再验证修正结果。

### 6.2 中期

**生产数据库连接与安全隔离：** 为小模型配置只读权限，禁止 `DELETE`、`UPDATE`、`DROP` 等危险操作，之后接入真实 MySQL 进行执行准确率评估，获得最贴近生产的评估结果。

**DAG 并行调度实现：** 大模型在分解任务时，输出结构需升级为带依赖关系的任务图：

```json
[
  {"id": 1, "task": "统计各项目收款总额", "depends_on": []},
  {"id": 2, "task": "统计各项目开票总额", "depends_on": []},
  {"id": 3, "task": "找出收款率低于30%的项目", "depends_on": [1, 2]}
]
```

**相关子查询改写训练：** 在训练数据中补充"相关子查询 → JOIN 改写"的样本，使模型天然倾向于输出 JOIN 形式，从根本上减少串行依赖，提升系统并行度。

### 6.3 长期

**生产数据飞轮：** 从生产环境持续收集自我纠错三元组，作为训练数据定期补充进训练集，形成模型与数据共同成长的正向循环。数据质量过滤策略：只收集"一次出错、下一次直接修正成功"的案例，多次反复才成功的案例质量不稳定，暂不纳入。

**Skills 渐进式 Schema 披露优化：** 上线后收集用户查询 → 实际使用的视图/字段映射（从执行日志反推），用真实数据微调 Schema Linking 组件，使大模型的视图筛选越来越精准。

**LLM-as-Judge 评估：** 对于语义正确性难以用执行结果量化的情况（如 NULL 处理、精度差异），引入大模型作为裁判进行语义级别的评估。

**多轮纠错训练：** 当前训练数据为单轮纠错（一次报错 → 一次修正）。积累生产数据后，可扩展为多轮对话训练，提升模型处理复杂错误链的能力。
